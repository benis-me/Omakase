grok-build × Omakase

它们都在让 AI 干活，但解决的是两层不同的问题。

一句话结论：grok-build 自己就是 coding agent；Omakase 编排的是外部 agent CLI。

两者各是什么

grok-build

grok-build/README.md 直接把它定义成 terminal-based AI coding agent；同一份仓库布局又把 xai-grok-shell 写成 Agent runtime，把 xai-grok-tools 写成工具实现，把 xai-grok-pager 写成 TUI。证据落点：grok-build/README.md。

Omakase

它识别并拉起外部 agent CLI，把每一轮调用接进 workflow；provider 负责单轮执行，engine 再在 harness 与 orchestrator 上组织 verify loop。它的重心是流程编排。证据落点：在 Omakase 源码中，README.md、packages/providers/src/runner.ts、packages/engine/src/harness.ts、packages/engine/src/orchestrator.ts。

三组关键对比

1. 谁真正掌握执行权

grok-build 的 Agent 是真实运行时对象，内部直接持有 ToolBridge、system_prompt、compaction policy 与 hosted tools；README.md 也把 xai-grok-shell 明写成 Agent runtime。证据落点：grok-build/crates/codegen/xai-grok-agent/src/agent.rs、grok-build/crates/codegen/xai-grok-tools/src/bridge.rs、grok-build/README.md。Omakase 则把执行权交给外部 provider，自身负责 workflow runtime 与 provider fallback。证据落点：在 Omakase 源码中，packages/providers/src/registry.ts、packages/providers/src/runner.ts。

2. tools 与 subagent 属于谁

grok-build 的工具是第一方内建实现，register_all() 会把 bash、read_file、grep、search_replace、task、web_search 等标准工具接进同一套 toolset；subagent resolution 也由单独 crate 处理。证据落点：grok-build/crates/codegen/xai-grok-tools/src/implementations/grok_build/mod.rs、grok-build/crates/codegen/xai-grok-subagent-resolution/src/lib.rs。Omakase 这一层更像是反复调用外部 agent，再把结果纳入 workflow；从本文列出的路径看，工具能力主要仍归属下层 CLI。

3. 产品重心放在哪一层

grok-build 的重心更偏向单个 agent 本体：同仓库、同产品闭环内同时覆盖 ACP 与原生 TUI。证据落点：grok-build/crates/codegen/xai-acp-lib/src/lib.rs、grok-build/crates/codegen/xai-grok-pager/src/lib.rs、grok-build/Cargo.toml。Omakase 的重心则更偏向 orchestration：verify、resume、crystallize 与 TUI 都围绕流程治理与复用展开。证据落点：在 Omakase 源码中，packages/engine/src/verify.ts、packages/engine/src/crystallize.ts、packages/tui/src/launch.ts。

各自更强在哪

grok-build 更强在单个 agent 的完整度：runtime、tools、subagent、ACP 与原生 TUI 都在一个 Rust 本体里。

Omakase 更强在 orchestration：它把已有 agent CLI 组织成可验证、可恢复、可沉淀为 workflow 的执行系统。

如果你要一个自己写、自己调工具、自己带终端界面的 agent，看 grok-build；如果你要一个把不同 agent 组织成稳定 workflow 的底座，看 Omakase。
